一、项目背景与选题意义

一、项目背景与选题意义

1.1 背景分析

在高校教学管理中，导师与学生之间的指导关系管理是一个典型而又繁琐的场景。以计算机学院为例，每年需要进行本科毕业设计导师分配、研究生导师双选、双创项目指导、学科竞赛指导以及社团活动指导等多种类型的师生配对。传统的人工管理方式大多依赖 Excel 表格和纸质审批，存在以下痛点：

  • 信息分散：导师信息、学生信息、配对关系、项目进展、成果产出分散在不同的表格或系统中，难以统一管理。
  • 配对效率低：师生双向选择、审核流程依赖邮件或线下沟通，周期长、容易遗漏。
  • 成果追溯困难：指导过程中产出的论文、奖项、证书、照片等资料缺乏系统化的归档和检索机制。
  • 权限混乱：管理员、导师、学生能看到的数据边界模糊，信息安全隐患大。
一、项目背景与选题意义

1.2 选题意义

本系统旨在构建一个基于 Web 的导师学生配对管理平台，核心目标如下：

  1. 统一信息管理：将导师、学生、配对、项目、成果五大数据域整合到一个系统中，打破信息孤岛。
  2. 规范配对流程：实现"申请 → 审核 → 配对通过 → 取消"的完整配对生命周期管理，状态流转清晰可控。
  3. 项目与成果归档：支持指导项目的创建与跟踪，以及论文、奖项、证书等成果资料的上传与展示，形成完整的指导档案。
  4. 细粒度权限控制：基于 RBAC（基于角色的访问控制）模型，管理员、导师、学生各司其职，数据隔离到位。

四、数据库设计

4.4 设计亮点

  1. 账号与身份分离：sys_user 负责登录认证，teacher / student 负责业务属性。一个用户账号通过 user_id 外键关联一个身份实体，实现了认证层与业务层的解耦。

  2. 多对多关联表：project_student 和 achievement_student 使用独立的关联表 + 联合唯一索引，支持一个项目/成果关联多个学生，避免了单字段逗号分隔的低效设计。

  3. 逻辑删除：导师、学生、项目、成果表均使用 MyBatis Plus 的逻辑删除功能（deleted 字段），删除操作不会真正移除数据，保证历史数据的完整性和可追溯性。

  4. 邮箱登录：采用 email 作为全局唯一登录凭证，方便用户记忆，同时 username 用姓名自动生成，重名按序追加数字后缀，友好又灵活。

  5. 增量迁移机制：DatabaseSchemaMigration 类在应用启动时自动检测并补全缺失的表/字段，支持从旧版本数据库平滑升级，无需手动执行 ALTER 脚本。

5.1 分层架构详解

后端严格遵循 Controller → Service → Mapper 三层架构，各层职责清晰：

Controller 层（8 个控制器）

每个控制器负责一个业务域的请求接入，统一使用 @RestController + @RequestMapping：

控制器 路由前缀 主要端点
AuthController /api/auth GET captcha, POST login, POST register/student, POST register/teacher
DashboardController /api/dashboard GET stats
UserController /api/user GET list, POST, PUT /{id}, DELETE /{id}
TeacherController /api/teacher GET list, GET /{id}, POST, PUT /{id}, DELETE /{id}
StudentController /api/student GET list, GET /{id}, POST, PUT /{id}, DELETE /{id}
MatchingController /api/matching POST create, GET list, PUT audit/{id}, PUT cancel/{id}
ProjectController /api/project POST, GET list
AchievementController /api/achievement POST upload(multipart), PUT /{id}, GET list, DELETE /{id}

Service 层（9 个服务接口 + 实现）

服务层是业务逻辑的核心，关键服务如下：

服务 核心职责
AuthService 验证码生成、邮箱密码认证、JWT 签发
RegisterService 学生/导师注册事务（sys_user + student/teacher 双表写入）
MatchingService 配对申请的创建（自动绑定当前用户身份）、列表查询（角色数据隔离）、审核与取消

AchievementService 成果上传（文件验证与存储）、编辑（权限校验）、删除（物理文件清理）、列表查询（计算 editable 标记）
SysUserService 管理员的用户 CRUD 操作
TeacherService / StudentService 导师/学生实体的 CRUD
DashboardService 五维度统计数查询

Mapper 层（9 个 Mapper 接口）

全部继承 MyBatis Plus 的 BaseMapper<T>，零 XML 配置实现基础 CRUD。其中两个 Mapper 包含自定义 SQL 注解：

  • ProjectStudentMapper：使用 @Select 注解实现项目-学生 JOIN 查询，用 <foreach> 动态 SQL 支持批量 ID 查询。
  • AchievementStudentMapper：同样使用 @Select + <foreach> 注解实现成果-学生关联 JOIN 查询。

5.2 统一响应格式

所有 API 返回统一封装为 Result<T>：

{
  "code": 200,
  "message": "success",
  "data": { ... }
}

分页接口额外封装为 PageResult<T>：

{
  "code": 200,
  "message": "success",
  "data": {
    "total": 100,
    "records": [ ... ]
  }
}

设计优势：前端无需判断不同接口的返回格式，统一通过 res.code === 200 判定请求是否成功，降低了前后端联调成本。

五、后端核心实现

5.3 全局异常处理

GlobalExceptionHandler 使用 @RestControllerAdvice 统一拦截所有异常：

异常类型 HTTP 状态码 处理逻辑
BusinessException 400 返回自定义错误码和消息
MethodArgumentNotValidException 400 提取第一个字段校验失败信息
MissingServletRequestParameterException 400 提示缺少的必要参数名
MaxUploadSizeExceededException 400 提示文件大小超限
BadCredentialsException 401 统一提示 "邮箱或密码错误"
AccessDeniedException 403 提示权限不足
Exception（兜底） 500 记录异常堆栈日志，返回通用错误

设计优势：Controller 代码中无需编写 try-catch 块，抛出 BusinessException 即可由全局处理器自动转换为标准 JSON 响应，代码更简洁。

对象类型 包路径 用途 举例
Entity entity/ 与数据库表一一映射，由 MyBatis Plus 管理 Teacher, Student, Matching
DTO dto/ 接收前端请求参数，带 JSR-303 校验注解 LoginDTO, ProjectDTO, StudentRegisterDTO
VO vo/ 返回给前端的视图对象，可包含跨表聚合字段 MatchingVO(含 teacherName、studentName), AchievementVO(含 editable、uploaderName)

设计优势：Entity 不直接暴露给前端，避免了敏感字段泄露（如 password）；VO 可以灵活组装多表聚合数据（如 MatchingVO 同时包含导师姓名和学生姓名的冗余字段），减少前端二次查询。

6.4 Axios 拦截器

请求拦截器：自动从 localStorage 读取 token 并添加到 Authorization: Bearer {token} 请求头。

响应拦截器：

  1. res.code !== 200 → 弹出错误提示；若 code=401 则清除本地存储并跳转登录页
  2. HTTP 状态码错误（如网络断开）→ 弹出错误提示；若 401 同样跳转登录

设计优势：所有 API 请求无需手动处理 token 注入和异常弹出，统一由拦截器完成，减少 8 个页面视图中的重复代码。

七、安全与权限控制

7.2 JWT 设计

JWT Token 的 Payload 包含：

{
  "sub": "user@example.com",   // 邮箱作为 subject
  "userId": 1,
  "username": "张三",
  "role": "STUDENT",
  "iat": 1719000000,           // 签发时间
  "exp": 1719086400            // 过期时间 (签发时间 + 86400000ms)
}
  • 签名算法：HMAC-SHA256，密钥从 application.properties 的 jwt.secret 读取
  • 有效期：24 小时
  • 无状态：服务端不存储 session，所有必要信息编码在 token 中
七、安全与权限控制

7.3 RBAC 权限模型

系统定义了三种角色常量：

ROLE_ADMIN   = "ADMIN"    // 超级管理员
ROLE_TEACHER = "TEACHER"  // 导师
ROLE_STUDENT = "STUDENT"  // 学生

权限控制分为两个层级：

1. 接口级权限（Spring Security @PreAuthorize）：

@PreAuthorize("hasRole('ADMIN')")                    // 仅管理员
@PreAuthorize("hasAnyRole('ADMIN', 'TEACHER')")      // 管理员或导师

2. 数据级权限（Service 层手动判断）：

  • 导师/学生只能查看与自己相关的配对、项目、成果
  • 导师只能编辑/删除自己上传的成果
  • 管理员拥有全量数据访问权

SecurityUtils 工具类提供了便捷的静态方法：

SecurityUtils.getLoginUser()   // 获取当前登录用户
SecurityUtils.isAdmin()        // 是否管理员
SecurityUtils.isTeacher()      // 是否导师
SecurityUtils.isStudent()      // 是否学生
七、安全与权限控制

7.4 密码安全

  • 使用 BCrypt 单向哈希算法，自动生成随机盐，同一密码每次加密结果不同
  • 即使数据库泄露，攻击者也无法还原原始密码
  • DataInitializer 启动时自动检测并重置默认账号密码为 123456

八、关键功能详解

八、关键功能详解

8.1 图形验证码

CaptchaUtil 自研验证码工具，不依赖任何第三方验证码库：

  • 从 31 个字符池（23456789ABCDEFGHJKLMNPQRSTUVWXYZ，剔除了易混淆的 0/O/1/I）中随机抽取 4 位
  • 使用 Java AWT Graphics2D 绘制 120×40 像素的 PNG 图片
  • 加入 6 条随机干扰线和 3 种随机颜色
  • 每个字符独立随机旋转 ±15°
  • 验证码存储在 ConcurrentHashMap 内存中，有效期 5 分钟，一次性消费（校验后立即删除）
  • 最终返回 Base64 编码的 Data URI，前端直接 <img src="data:image/png;base64,..."> 展示
八、关键功能详解

8.2 邮箱登录 + 自注册

系统摒弃了传统的用户名密码登录，改用 邮箱作为全局唯一登录凭证：

  • 注册：学生/导师通过公开接口自行注册，系统自动在 sys_user 表创建用户（username=姓名，重名自动追加序号）并在 student/teacher 表创建身份记录
  • 登录：输入邮箱 + 密码 + 验证码，Spring Security 的 DaoAuthenticationProvider 通过 UserDetailsServiceImpl 加载用户
  • 重名处理：RegisterServiceImpl.resolveUsername() 方法会循环检测 sys_user 中是否存在同名用户，存在则自动追加序号（张三 → 张三1 → 张三2...）
八、关键功能详解

8.3 配对生命周期管理

配对功能是本系统的核心业务，实现了完整的状态机：

创建申请 (PENDING)
    │
    ├── 管理员审核通过 (APPROVED) → 记录 startTime → 建立正式指导关系
    ├── 管理员审核拒绝 (REJECTED) → 流程结束
    └── 创建者取消 (CANCELLED) → 记录 endTime
        │
APPROVED 状态也可以被取消 (CANCELLED)

权限细节：

  • 创建配对时，系统根据当前登录用户角色自动绑定导师方或学生方身份
  • 导师角色自动填入 teacherId = user.getTeacherId()，无需前端传递
  • 学生角色同理自动填入 studentId = user.getStudentId()
  • 管理员可指定任意师生组合
  • 取消配对的权限：管理员可取消任意配对；导师和学生只能取消涉及自己的配对
八、关键功能详解

8.4 多对多项目与成果

本系统支持一个项目/成果关联多名学生：

数据存储：

  • project 表保留 student_id 字段作为主学生（兼容历史数据）
  • project_student 关联表以 (project_id, student_id) 唯一索引存储多对多关系
  • achievement / achievement_student 同理

数据一致性校验：

  • 创建项目/成果时，系统会校验所选学生的 student_id 必须在 matching 表中存在与该导师的 APPROVED 状态记录
  • 即：只有已配对通过的学生才能被导师关联到项目或成果中

列表查询的智慧回退：

  • 优先从 project_student 关联表 JOIN 查询学生信息
  • 若关联表不存在或无数据，自动回退到 project.student_id 字段
  • 这种设计保证了向后兼容性（从单学生到多学生的平滑演进）
八、关键功能详解

8.5 成果文件管理

上传：

SecurityUtils.saveFile(file, uploadPath)
  1. 校验文件扩展名（仅 6 种白名单）
  2. 使用 UUID 重命名文件，防止文件名冲突和路径穿越
  3. 存储到配置的 file.upload.path 目录（默认 upload/）
  4. 返回 URL 路径如 /upload/a1b2c3d4.pdf
  5. Spring MVC 通过 WebMvcConfig 映射 /upload/** 到本地目录，实现文件直接访问

编辑：

  • 重新上传文件时，自动删除旧文件（物理删除）
  • 编辑仅限上传者本人或管理员

删除：

  • 管理员可删除任意成果
  • 普通用户只能删除自己上传的成果
  • 删除时自动清理关联的物理文件

editable 标记： 成果列表接口返回 editable: true/false，前端根据该字段决定是否展示编辑/删除按钮，避免了前端重复判断权限。

八、关键功能详解

8.6 数据库增量迁移

DatabaseSchemaMigration 在应用启动时自动执行（@Profile("!test") 排除测试环境）：

  1. ensureProjectStudentTable()：若 project_student 表不存在则创建；自动将 project 表中 student_id IS NOT NULL 的数据迁移到关联表
  2. ensureAchievementUserIdColumn()：检测 achievement 表是否有 user_id 字段，若无则 ALTER TABLE 添加；自动从 teacher.student 反向填充 user_id
  3. ensureAchievementStudentTable()：同理创建 achievement_student 关联表并自动迁移数据

设计价值：这是生产环境部署的"零停机升级"方案。老版本数据库无需手动执行 SQL，应用启动时自动完成结构升级和数据迁移。
